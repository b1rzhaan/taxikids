# KidsTransfer («Детское такси») — полное описание проекта

Сервис безопасной перевозки детей. Родитель заказывает поездку ребёнка, штатный
водитель компании выполняет её, а оператор/админ/бухгалтер управляют всем через
веб-кабинет. Три приоритетных требования — **карта и маршруты**, **интеграция с
банком** и **основное приложение** — реализованы как заменяемые модули (ports &
adapters), чтобы менять провайдера карты или банка без переписывания системы.

---

## 1. Технологический стек

| Слой | Технологии |
|---|---|
| **Backend** | Python, Django 5, Django REST Framework, JWT (SimpleJWT), Django Channels + Redis (live-трекинг по WebSocket), Celery (регулярные поездки), PostgreSQL (в Docker) / SQLite (локально) |
| **Веб-кабинет (сайт)** | Next.js 14 (React), TypeScript, Tailwind CSS, SWR, Leaflet + OpenStreetMap |
| **Мобильное приложение** | Flutter (Dart), provider, dio (JWT + авто-refresh), google_fonts (Manrope), flutter_map, 2GIS MapGL JS (WebView), geolocator, flutter_tts (озвучка), web_socket_channel |
| **Инфраструктура** | Docker Compose (Postgres + Redis + Backend + Web), одна команда запуска |

## 2. Архитектура

### Роли (RBAC)
`parent` (родитель), `driver` (водитель), `operator`, `admin`, `accountant`.
Роль зашита в JWT-claims; доступ ограничен DRF-permissions (`IsParent`, `IsDriver`,
`IsOperator`, `IsAdmin`, `IsAccountant`, `IsStaffRole`).

### Модули backend (`backend/apps/`)
- **accounts** — пользователи, профили, JWT-авторизация, RBAC.
- **children** — дети (имя, класс, школа, «основной», особенности/комментарий).
- **drivers** — водители, машины, тарифы, схемы зарплат, онлайн-статус.
- **trips** — заказы, **стейт-машина поездки**, история статусов, локации, регулярные планы, оценки.
- **maps** — интерфейс `MapProvider` + адаптеры (Mock, 2GIS): геокодинг, подсказки, маршруты.
- **payments** — интерфейс `PaymentProvider` + адаптеры (Mock, Halyk ePay).
- **wallet** — кошелёк, транзакции, пополнение через банк, абонементы.
- **payouts** — начисления водителям, выплаты, отчёты.
- **notifications** — уведомления, FCM-токены, экстренная связь (SOS).
- **statistics** — дашборд, выручка, эффективность водителей.

### Принципы
- **Ports & Adapters (гексагон)** для внешних сервисов — карта и банк за интерфейсами.
- **Стейт-машина** — единственный источник правды о статусах поездки (переходы валидируются по роли + текущему статусу, пишется аудит).
- **Тонкие вьюхи, толстые сервисы** — бизнес-логика в `services.py`.

## 3. КАРТА И МАРШРУТЫ

### Архитектура (заменяемая)
В `apps/maps/providers/` описан интерфейс **`MapProvider`** с методами:
`geocode` (адрес→координаты), `suggest` (автокомплит), `reverse_geocode`
(координаты→адрес), `route` (маршрут + расстояние + время + пробки + геометрия).
Провайдер выбирается переменной окружения `MAP_PROVIDER` (`mock` | `twogis`).
Есть `MockMapProvider` (работает офлайн, без ключа) для разработки/демо.

### Что используется СЕЙЧАС — 2GIS Platform (REST) + 2GIS MapGL JS
1. **Данные и маршруты — 2GIS Platform API** (по реальному ключу, демо-тариф):
   - **Geocoder API** — адрес → координаты.
   - **Suggest** — подсказки адресов при вводе (со смещением к городу — Алматы, чтобы «Абая 45» не отдавало другие города).
   - **Reverse** — тап по карте → точный адрес (параметр `type=building`, иначе 2GIS отдаёт только район).
   - **Routing API 7.0.0** — маршрут **с учётом пробок** (`traffic_mode=jam`), расстояние, время, геометрия (полилиния 180+ точек).
   - Проверено вживую: маршрут ≈ 5.5 км / 15 мин с пробками.
2. **Отрисовка карты:**
   - **Веб-кабинет** — Leaflet + OpenStreetMap (карта активных поездок у оператора).
   - **Flutter (трекинг, заказ, экран водителя)** — `flutter_map` + тайлы CartoDB Voyager; при этом **сам маршрут строится реальным 2GIS** и рисуется поверх тайлов.
   - **Навигатор водителя** — **2GIS MapGL JS** внутри WebView: настоящая карта 2GIS с 3D-наклоном, поворотом по курсу (course-up), машинкой-PNG, исчезающим пройденным «хвостом» и **голосовыми подсказками** («через 200 м поверните направо»). MapGL JS входит в демо-ключ 2GIS без лимита.

### Почему 2GIS
Лучшее покрытие, геокодинг и данные о пробках по Казахстану/СНГ; MapGL даёт 3D-навигацию.

### Что будет в БУДУЩЕМ
- **Нативный 2GIS Mobile SDK** (`dgis_mobile_sdk_full`) для «настоящего» навигатора уровня Яндекс.Навигатора (полный turn-by-turn, офлайн-карты, голос). Требует **отдельный мобильный ключ `dgissdk.key`** (не входит в демо-ключ, обычно платный по договору). Интеграция уже прототипировалась; сейчас используется MapGL JS через WebView, т.к. он работает с текущим ключом.
- Замена тайлов CartoDB на 2GIS MapGL и в остальных экранах (единый стиль).
- Полный переход live-трекинга на **WebSocket** (Django Channels уже реализован) вместо periodic-опроса.
- Смена провайдера (на Яндекс/Google) — только добавить адаптер, менять одну строку `MAP_PROVIDER`.

## 4. ИНТЕГРАЦИЯ С БАНКОМ

### Архитектура (заменяемая)
В `apps/payments/providers/` описан интерфейс **`PaymentProvider`**:
`create_payment` (создать платёж/виджет банка), `verify_webhook` (проверить
подпись колбэка), `refund`. Провайдер выбирается `PAYMENT_PROVIDER`
(`mock` | `halyk`). Есть `MockPaymentProvider` (эмуляция карточной формы) для разработки.

### Что используется СЕЙЧАС — Halyk Bank ePay (тестовый контур)
Реальная интеграция с банком **Halyk (Народный банк)** через ePay:
1. Backend получает **OAuth-токен** у Halyk (endpoint `epay-oauth`), формирует
   платёжный объект (invoiceId, сумма, backLink/failureBackLink, terminal, токен).
2. Приложение открывает **хостовую карточную форму Halyk** — виджет `payment-api.js`
   в WebView. Важно: страница виджета **рендерится нашим backend** (реальный origin),
   иначе банковский скрипт не инициализируется (был баг «белый экран» при
   `about:blank`).
3. После оплаты банк редиректит на `backLink` (успех) / `failureBackLink` (ошибка);
   приложение это ловит и **подтверждает платёж на сервере**. Плюс `postLink`
   (webhook) для асинхронного подтверждения и **идемпотентность** (без задвоения).
4. Так оплачиваются **и поездки, и пополнение кошелька** — обе операции идут через Halyk.

Мок-провайдер остаётся как запасной (`PAYMENT_PROVIDER=mock`) — своя «страница банка» с формой карты.

### Что ещё построено вокруг денег
- **Кошелёк** родителя: пополнение через банк, списания, история операций.
- **Абонементы** (месячные, N поездок) — покупка с баланса.
- **Начисление водителю** при завершении поездки (сплит 70/30: водителю 70%, приложению 30%; настраивается схемой зарплаты).
- **Выплаты**: водитель жмёт «Вывести» → создаётся заявка на выплату; бухгалтер видит и подтверждает; отчёты + экспорт CSV.
- Бухгалтер видит все платежи/транзакции; статистика считает выручку, расход водителям и доход приложения.

### Что будет в БУДУЩЕМ
- Переключение Halyk с **тестового на боевой** контур (нужен договор мерчанта и боевые ключи) — код не меняется, только настройки.
- Добавление других банков/эквайеров (**Kaspi**, CloudPayments и т.д.) как новых адаптеров `PaymentProvider` — без переписывания приложения.
- Проверка подписи вебхука боевыми сертификатами Halyk.

## 5. ОСНОВНОЕ ПРИЛОЖЕНИЕ

Одно Flutter-приложение с двумя режимами — родитель и водитель — маршрутизация по
роли из JWT. Дизайн — жёлтый фирменный стиль «Детское такси», шрифт Manrope,
мягкие карточки, дашборды.

### Родитель
- **Главная (дашборд)**: приветствие, «Мои дети», поездки на сегодня, текущая поездка (ETA), быстрые действия.
- **Заказ поездки**: карта сверху (откуда/куда с поиском адресов 2GIS и сохранением в избранное), выбор ребёнка, дата/время, стоимость с учётом пробок, «Заказать».
- **Оплата**: Halyk ePay (карточная форма банка).
- **Трекинг**: карта с маршрутом, **степпер этапов** (Заказ → Водитель → В машине → Доставлен), карточка водителя (звонок), live-позиция машины.
- Поездки (история), Финансы (кошелёк/абонементы), Сообщения/SOS, Профиль. Оценка водителя после поездки.

### Водитель
- **Главная**: тумблер «На линии», заработок и число поездок за сегодня, следующая поездка, быстрые действия.
- **Поездки**: вкладки «Текущие» и «Доступные» (водитель сам принимает оплаченный заказ).
- **Навигатор**: 2GIS MapGL, ведение по маршруту, машинка едет, **голосовые подсказки**, вкл/выкл озвучки в настройках.
- **Заработок**: периоды (Сегодня/Неделя/Месяц), баланс + «Вывести», история начислений.
- Профиль (авто, документы, статистика, поддержка). Оценка заказа после завершения.

### Ядро логики
- **Стейт-машина поездки** — 11 статусов: `created → waiting_payment → paid →
  driver_assigned → driver_on_way → driver_arrived → child_picked_up → in_progress →
  child_delivered → completed` (+ `cancelled`). Каждый переход разрешён только
  определённым ролям, пишется история (аудит) и триггерит уведомление.
- **Регулярные поездки** (шаблон-расписание генерирует поездки).
- **Уведомления** по статусам (in-app + FCM-стаб), **экстренная связь (SOS)**.
- **Оценки** с обеих сторон, рейтинг водителя пересчитывается.
- **Live-трекинг**: водитель шлёт GPS → родитель видит движение (сейчас опрос, готов WebSocket через Channels).

### Веб-кабинет (сайт)
Для оператора/админа/бухгалтера: логин по ролям, дашборд («Панель владельца»:
выручка / расход водителям / доход приложения 70/30), заказы + назначение водителя,
**карта активных поездок**, водители (одобрение документов), платежи, выплаты (+CSV),
статистика.

## 6. Как запускается

Всё поднимается одной командой через Docker:
```
start-docker.bat     # Postgres + Redis + Backend + Сайт
```
- Сайт (кабинет): http://localhost:3000
- API / Swagger: http://localhost:8000/api/docs/
- Django admin: http://localhost:8000/admin/
- Демо-логины: `parent@kids.kz`, `driver@kids.kz`, `operator@kids.kz`,
  `admin@kids.kz`, `accountant@kids.kz` — пароль вида `<роль>12345`.

Мобильное приложение (Flutter) запускается из Android Studio и ходит на тот же
backend (эмулятор → `10.0.2.2:8000`). Остановить стек: `stop-docker.bat`.

| Требование | Сейчас | В будущем |
|---|---|---|
| **Карта** | 2GIS Platform (геокодинг, подсказки, маршруты с пробками) + 2GIS MapGL JS навигатор (3D, голос); отрисовка на flutter_map/Leaflet | Нативный 2GIS Mobile SDK (полный turn-by-turn), WebSocket-трекинг, единые 2GIS-тайлы |
| **Банк** | Halyk ePay (тест): OAuth + хостовая карта в WebView + webhook + идемпотентность; поездки и кошелёк | Боевой Halyk; доп. эквайеры (Kaspi и др.) как новые адаптеры |

Ключевая ценность архитектуры: **и карту, и банк можно заменить сменой одной
настройки и добавлением адаптера — без переписывания приложения.**
